# Review: `ui/components`

1. Tighten the shared type plan so it matches the analyzed data shapes instead of hard-coding narrower object-only task collections.
The analysis explicitly allows `tasks` on job summaries and job details to be either an object map or an array, but the spec narrows both to `Record<string, TaskStateObject>`. That is a behavioral change at a core boundary and will either force hidden adapter work or make the migration reject valid existing data. The spec should either preserve the union at the component boundary or state exactly where normalization to object maps is guaranteed before these components receive props.

2. Define the time-store subscription contract precisely enough for strict TypeScript and leak-free React usage.
The analysis says `LiveText` and `TimerText` depend on `subscribe`, `getSnapshot`, `getServerSnapshot`, `addCadenceHint`, and `removeCadenceHint`, but the spec only turns those into a high-level dependency mention and acceptance criteria about `useSyncExternalStore`. It should state when cadence hints are registered and removed, whether multiple components sharing a cadence coalesce cleanly, and what snapshot shape is consumed. Without that, the TS rewrite can satisfy the rendering criteria while regressing the analyzed no-per-component-polling lifecycle behavior.

3. Reconcile the test-environment behavior with the analyzed browser-only assumptions instead of introducing a `process.env.NODE_ENV === "test"` branch.
The analysis says these components assume a browser environment and use DOM APIs like `ResizeObserver`, `requestAnimationFrame`, and `matchMedia`. The implementation spec adds acceptance criteria requiring `DAGGrid` and `PipelineDAGGrid` to skip those APIs entirely in test mode. That is a new environment-specific code path that is not derived from the source behavior and risks masking real layout bugs. The spec should prefer injectable geometry helpers or test doubles rather than baking a test-only runtime branch into production code.

4. Restore the analyzed `TaskDetailSidebar` export contract instead of reducing it to a single default component shape.
The analysis calls out `TaskDetailSidebar` as both a named export and a memoized default export. The implementation file list and type plan only describe one component file and do not require preserving both export forms or the memoized public surface. If other modules import the named version today, the rewrite can break them while still meeting the current acceptance criteria. The spec should make the export surface explicit for all dual-export components called out by the analysis.

5. Specify the SSE streaming and teardown mechanics in `TaskCreationSidebar` more concretely, because the current spec is too loose for a fragile boundary.
The analysis describes fetch-based SSE streaming with incremental decoding, proposal extraction, and request-phase gating, while the spec only says to stream from `/api/ai/task-plan`, disable input during three phases, and register `beforeunload` when messages exist. It should also define event framing expectations, abort behavior on close/unmount, how partial chunks are buffered across decoder boundaries, and whether stale streams are ignored when a new send begins. Otherwise the rewrite can easily leak readers or mis-parse streamed responses without violating the current acceptance criteria.

6. Make the markdown and file-preview rendering rules explicit enough to preserve observed behavior and security posture.
The analysis distinguishes between `MarkdownRenderer` with GFM/custom overrides and `TaskFilePane` with basic Markdown, JSON pretty-printing, plain text, and binary placeholders. The spec lists those capabilities, but it does not define whether raw HTML is disabled, how unknown MIME types fall back, or whether large files are truncated before rendering. Those details matter for determinism and browser safety in a UI that renders server-provided content. The review should require explicit rendering rules rather than leaving them implicit.

7. Clarify the delayed-navigation contract in `AddPipelineSidebar` so the rewrite does not lock in an arbitrary timer without observability.
The analysis notes the 1-second post-create delay exists to let a backend watcher detect the registry change, and the spec preserves that delay verbatim. What is missing is the surrounding contract: what response fields are required from `POST /api/pipelines`, what happens if navigation target data is still unavailable after the delay, and whether the pending state stays visible during the wait. The spec should document those details so the preserved delay remains deterministic and user-visible rather than a hidden magic timeout.
