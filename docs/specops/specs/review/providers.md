# Review: `providers`

1. Define the streaming contract all the way through the public API instead of only at the DeepSeek adapter.
The spec adds `stream?: boolean` to `ChatOptions`, but `ChatResponse`, `HighLevelLLM.chat()`, `complete()`, and `ConversationChain.execute()` are all typed as non-streaming `Promise<ChatResponse>`. Later, Step 6 gives `deepseekChat()` a union return with an async generator, but Step 9 still says `chat()` always returns `ChatResponse`. The review should require one explicit rule: either streaming is adapter-only and never exposed through `src/llm/index.ts`, or the gateway/high-level API types and acceptance criteria must also describe the streaming return path.

2. Reconcile the `tryParseJSON()` contract with the required `ProviderJsonParseError` behavior.
Step 2 says `tryParseJSON()` returns the original text on total parse failure, while the analysis and acceptance criteria say adapters throw `ProviderJsonParseError` when provider responses are not valid JSON. Those two contracts leave an implementation gap: an adapter cannot tell whether a returned string is a legitimate non-JSON text result or a failed JSON parse without extra rules. The spec should state exactly which layer converts parse failure into `ProviderJsonParseError` and how adapters distinguish text mode from JSON-mode failure.

3. Preserve the analyzed `complete()` default-provider behavior or mark the change explicitly.
The analysis calls out that `complete()` currently defaults to `"openai"` in test mode instead of using `getConfig().llm.defaultProvider`, but acceptance criterion 10 rewrites the contract to always use the configured default provider. Because the review prompt says not to change original scope or intent based on the analysis, the spec should either keep the existing test-mode behavior for compatibility or clearly mark this as an intentional breaking cleanup and update the acceptance criteria/tests accordingly.

4. Do not change the `/tmp/messages.log` behavior implicitly under the guise of migration notes.
The Notes section intentionally gates the existing unconditional debug write behind `LLM_DEBUG`, but the analysis treats the current write as real behavior and the review prompt says not to change original scope or intent. That may be a good improvement, but it is still a behavioral change with observability and compatibility impact. The spec should either preserve the analyzed behavior for the migration or promote the new gated behavior into an explicitly approved requirement with matching acceptance criteria and tests.

5. Tighten the public typing around mock providers and provider-grouped callables so the factory APIs are actually implementable under `strict` TypeScript.
`createLLM()`, `createNamedModelsAPI()`, `createHighLevelLLM()`, and `registerMockProvider()` are central exports, but the proposed types mostly collapse them to `[provider: string]: unknown` or a mock object with a single `chat()` method. That is too weak to validate the analyzed contract that factory-built provider/model functions are callable and overrideable by `createLLMWithOverride()`. The spec should require concrete callable shapes for provider groups and mock providers; otherwise the migration can satisfy the loose types while still drifting from the existing API surface.
