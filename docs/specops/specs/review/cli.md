# Review: `cli`

1. Reconcile the planned `.ts` scaffold output with the rest of the project before implementation starts.
The analysis says the existing CLI scaffolds `tasks/index.js` and task modules with `.js` paths, while this spec intentionally changes them to `.ts` in acceptance criteria 24 and 29 and in Step 6. That is not just an internal refactor: it changes the on-disk workspace contract that pipeline loading depends on. The spec should either preserve the existing `.js` scaffold format or explicitly require the corresponding loader, registry, packaging, and runtime import changes everywhere that consumes these generated files. Without that, the rewrite can produce scaffolded pipelines that the rest of the system cannot execute.

2. Define how the executable entrypoint and package metadata are updated for a TypeScript CLI.
This spec creates `src/cli/index.ts`, but the current package contract still points the published binary and scripts at `src/cli/index.js` (`package.json` `bin.pipeline-orchestrator` and `scripts.analyze`). The implementation plan never says whether the package metadata will be updated, whether a `.js` shim will remain, or whether Bun is expected to execute `.ts` directly after publish. That omission makes the migration incomplete at the packaging boundary. The spec should add one explicit requirement covering the installed executable path and any package-script updates.

3. Remove the unresolved `PipelineOrchestrator` API ambiguity from the implementation plan.
Acceptance criteria 18 requires `status` to create a `PipelineOrchestrator` with `{ autoStart: false }`, but the Notes section also says the migrated API may instead expose a factory and that this “will depend on the API module's migration.” That leaves a core dependency undefined inside the spec itself. The review should require one exact construction contract for the CLI to call, or define a thin adapter in the CLI layer, so implementation is not blocked on guesswork.

4. Make the `PO_UI_PORT` environment contract explicit instead of relying on `undefined` behavior.
The spec says `start` must spawn the UI child with `PO_UI_PORT: undefined` in the environment. In practice, process environment maps are string-based, and the effect of passing `undefined` through `Bun.spawn` is an implementation detail: it may omit the variable or coerce it unexpectedly. If the absence of `PO_UI_PORT` matters, the spec should say “omit this key from the child env.” If the literal string matters, it should say that directly. Right now the contract is ambiguous.

5. Add one exact parsing contract for `tasks/index.ts`; “regex-based parsing” is too underspecified to be reliable.
The spec replaces `eval()` with regex parsing, but it never constrains the file format tightly enough for that to be deterministic. Acceptance criterion 29 just says “parse the existing default export,” which leaves unclear whether comments, trailing commas, single quotes, multiline values, or non-generated formatting must be preserved or rejected. The spec should narrow the supported file shape and define the failure behavior when the file falls outside that shape. Otherwise the rewrite risks fragile parsing in a command that mutates user-visible workspace files.
