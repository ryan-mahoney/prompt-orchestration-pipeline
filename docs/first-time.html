First-Time User
Experience Guide

How to design every page so that first-time users achieve their first meaningful outcome as quickly as possible. Based on Samuel Hulick's Elements of User Onboarding framework, applied to Prompt Pipeline.

Companion to design-system.html v1.0 · March 2026

0. The Core Mindset

Before any pattern or component, internalize the philosophy that drives every onboarding decision.

Sell the outcome, not the tool

Users don't want "a pipeline orchestrator." They want to generate competency models, job frameworks, and talent structures in minutes instead of weeks. Show the fireball-throwing version of them — not the fire flower.

Attention is a leaking spacesuit

Every field, every click, every moment of confusion is air escaping. Minimize friction relentlessly. If a step isn't moving the user closer to their first successful pipeline run, cut it or defer it.

The product introduces itself

Don't layer an interface on top of the interface. The best onboarding is indistinguishable from the product. Prepopulated data, contextual hints, and smart defaults teach by doing — not by lecturing.

Quick wins build momentum

Get the user to a meaningful result within 5 minutes. For Prompt Pipeline, that means: upload a seed → see the pipeline graph → watch the first step complete. That first "DONE" badge is the hook.

2. Empty States

The most underused onboarding surface. Every page a new user visits will be empty. That emptiness is an opportunity to teach, motivate, and direct action.

Hulick: "Basecamp pre-populates fake projects so you know where they'll be and how they'll look. The interface uses itself to introduce itself."

A. Actionable Empty State (primary)

Use when the user needs to create something to see value. The empty state explains what goes here, why it matters, and gives one action.

No pipelines yet

A pipeline transforms a seed file into a complete job framework — levels, competencies, tasks, and attributes — in one run.

Key elements: (1) Icon that represents the missing content, not a generic ø. (2) Title names the thing ("No pipelines yet"). (3) Description explains what pipelines do in one sentence — outcome, not mechanics. (4) Primary CTA is the same green button from the nav ("Upload seed file"). (5) Secondary link offers a zero-effort alternative — the sample pipeline.

B. Prepopulated / Sample State

Use to show what the interface looks like with real data. Pre-fill with a demo pipeline so users understand the graph, node cards, and status badges before creating their own.

C. Empty State Inventory

Every page in the app must have a designed empty state. Here's the audit:

Page Empty state type Primary CTA Pipelines list Actionable + sample pipeline Upload seed file Pipeline detail N/A (always has graph) — Step output Contextual — "Step hasn't run yet" Start / Restart Seed library Actionable + templates Upload seed file Usage / billing Informational — "No usage yet" Run your first pipeline Search results No-results message Clear filters
Do Every empty state has: (1) what goes here, (2) why it matters, (3) one action to fix it. Offer a sample/template as a zero-friction alternative. Use the "Sample data" badge to mark demo content. Remove sample content once the user creates their first real item.
Don't Blank pages with just "No data." Illustrations or mascots (violates functionalist principles). Feature lists in empty states — describe the outcome, not the tool. Leave any page undesigned for the zero-data case.

3. Progress & Checklists

Show users where they are and what's left. Progress indicators reduce anxiety ("Am I doing this right?") and leverage the endowed progress effect to sustain motivation.

Hulick: "Start people off with some progress already completed. The Endowed Progress Effect gives people a psychological push forward." — Quora starts its checklist with 3/5 items pre-checked.

A. Getting Started Checklist

Persistent sidebar widget that tracks the user's first-run milestones. Starts with 1 item pre-completed (account creation) to trigger endowed progress.

Getting started 2 / 5
  • Create account
  • Upload a seed file
  • Run your first pipeline
    Click "Start" on any pending step
  • Review pipeline output
  • Export or iterate
Endowed progress: "Create account" is pre-checked — the user already has momentum. The active step has a highlighted ring and helper subtext telling them exactly what to do next. Completed items use strikethrough + muted color to fade into the background. The progress bar fills with brand green, providing a visceral sense of advancement. The whole widget is dismissable and collapses after all 5 items are done.

B. Setup Stepper (multi-step flows)

Use for linear multi-step processes like seed configuration or pipeline settings. Shows where the user is in the sequence.

Upload seed
2
Configure
3
Run pipeline
Spec: 28px circles, 2px connector lines. Done steps: green fill + checkmark. Active: accent ring + bold label. Future: gray border + muted label. Step labels are verbs ("Upload", "Configure", "Run") not nouns. Max 5 steps — if your flow has more, simplify the flow.

C. Pipeline Runtime Progress

The pipeline graph itself is a progress indicator. As each step completes, it transitions from PENDING → RUNNING → DONE, giving the user continuous visual feedback.

DONE — green fill, token/cost metadata appears
RUNNING — pulsing amber indicator
PENDING — gray, waiting for upstream
Why it works: Users can watch their pipeline "light up" from left to right as each step completes. This is inherently satisfying and eliminates the anxiety of "is it working?" The graph is the progress bar — no separate loading indicator needed.
Do Pre-complete the first checklist item (endowed progress). Show numeric progress ("2 / 5") alongside the visual bar. Make each checklist item clickable — it navigates to the relevant page or action. Auto-dismiss the checklist after all items are done, with a brief celebration.
Don't More than 5 checklist items. Progress bars without numeric labels. Checklists that block the UI (they should be a sidebar widget or collapsible card). Percentage-based progress for a 3-step flow (just show the steps).

4. Contextual Guidance

Teach users at the moment they need to know, not in a front-loaded tour. Guidance should be non-interruptive, dismissable, and embedded in the interface — not layered on top of it.

Hulick: "Don't try to fix a broken interface by overlaying yet another interface on top of it. Focus on better design rather than wasting time with tooltips or coach marks."

A. Inline Hint Banners

The primary guidance pattern. Placed inline within the page content, not floating. Dismissed permanently per-user. Use sparingly — max 1 per page.

Steps run left to right. Each step waits for its upstream dependencies to finish before it can start. Click any step card to see its output and configuration.
Your seed file is ready. Click "Rescan" to re-run the pipeline from the failed step, or "Restart" on any individual step card.
Two variants: Info (accent-50 background, ⓘ icon) for general guidance. Action (brand-50 background, ⚡ icon) for context-specific prompts tied to the current state. Both have: bold lead sentence, supporting text, × dismiss. Dismissed hints are stored per-user and never return.

B. Contextual Tooltips

Use for explaining a specific element the first time a user encounters it. Triggered by hover or first-time visibility — never as a sequential tour. One tooltip at a time.

Each step shows its token usage and cost after it completes.
Experts-in-job DONE
11.4k tokens · $0.0305
Spec: Dark background (gray-900), white text, 13px, max-width 260px. 6px arrow. "Got it" dismiss link. Shown once per element per user. Position: prefer top, fall back to bottom/left/right if clipped. Z-index: 100 (dropdown level). Entrance: fade-in 150ms.

C. Hotspot Beacons

Subtle pulsing dots that draw attention to a feature without interrupting. Click to reveal a tooltip. Use for one feature discovery moment per page, max.

← pulsing beacon on first visit
Spec: 12px accent-700 dot, positioned at top-right of the target element. Outer ring pulses at 2s interval (ease-in-out, scale 1→1.5, opacity 0.4→0). Click opens tooltip. Dismissed permanently after click. Max 1 beacon visible at a time — queue them if multiple features need discovery.

Pattern Priority

Choose the least intrusive pattern that gets the job done:

1st Smart defaults No guidance needed — the product just works. Pre-fill inputs, auto-select the right option. 2nd Empty states Teach through the absence of data. Explain what goes here and how to fill it. 3rd Inline hint banners Persistent, embedded guidance. Dismissable. 1 per page max. 4th Hotspot beacons Subtle attention-draw for feature discovery. Click-to-reveal. 5th Tooltips Targeted explanation on first encounter. One at a time. Last Product tours Sequential tooltip tours are a last resort. They interrupt flow, overwhelm memory, and feel coercive. Avoid if possible.
Do Favor inline guidance (hint banners, empty states) over floating guidance (tooltips, tours). Always make guidance dismissable. Store dismiss state per user. Tie guidance to context — show it when the user reaches the relevant page/state, not at sign-up.
Don't Sequential tooltip tours that hijack the interface. More than 1 hint banner per page. Tooltips that appear on hover without a dismiss — they'll fire on every visit. Coach marks on obvious UI elements ("This is the navigation bar"). Any guidance that blocks the user from interacting with the product.

5. Success States & Celebration

Celebrate the user's achievement, not your product's features. Success reinforcement confirms that the user made the right choice and motivates the next action.

Hulick: "When users finish the task, it's important to be there with positive reinforcement. Celebrate the success that you're helping someone achieve in their life."

A. First Pipeline Complete

A one-time celebration when the user's first pipeline reaches 100% DONE. This is the critical "aha moment" — the user has seen value.

Pipeline complete

9 steps finished in 34m 12s. Your competency model, levels, and job tasks are ready to review.

Tone: Calm and factual. "Pipeline complete" — not "Amazing! 🎉 You did it!" This is a professional tool; celebrate with data (steps, duration, output), not confetti. The CTA moves forward ("Review output") rather than looking backward. The secondary action lets them build momentum with another run.

B. Checklist Completion

When all getting-started items are done, transform the checklist into a brief success message, then auto-collapse after 5 seconds.

All set
You've completed the getting started guide.

C. Inline Step Success

When an individual pipeline step completes, the node card transitions smoothly: badge flips to green "DONE", token/cost metadata fades in, and the "Restart" button replaces "Start".

Do Celebrate with data: "9 steps, 34m 12s, $0.14 total cost." Always provide a forward-looking CTA ("Review output", "Run another"). Make the first success moment special (one-time modal) but keep subsequent completions inline and subtle. Auto-dismiss celebrations after a timeout.
Don't Confetti, emojis, exclamation marks, or animated mascots. Celebrations that block the user from continuing. "Great job!" tone — this is a power tool, not a gamified app. Celebrating partial completion ("3 of 9 steps done!") unless the user triggered a specific action.

6. Progressive Disclosure

Show only what the user needs right now. Reveal complexity as they advance. This reduces cognitive load for new users while preserving power for experienced ones.

Hulick: "If you look at your product as a user going from 0 to 60, you're designing for the person at 60. A majority of folks never reach that cool spot." — Design for the journey, not the destination.

A. Layered Interface

Map features to user maturity levels. New users see the essential view; power features reveal as they gain experience.

Stage Visible Hidden until needed First visit Pipeline graph, status badges, Upload Seed CTA, getting-started checklist Cost breakdown, token details, advanced config, API access, export options First pipeline run + Step output drawer, Restart/Start buttons, duration and cost per step Seed editor, step config overrides, batch operations Returning user + Pipeline list with history, total cost column, search and filters API keys, webhook config, team settings Power user + Full config, API playground, bulk actions, custom step definitions Nothing — all features accessible

B. "Show more" Patterns

Within a single page, use expandable sections to hide advanced options behind a clear affordance.

Collapsed (default for new users)
Expanded (on click)
Spec: Toggle is a text button with +/− icon, accent-700 color. Expanded content has a 2px left border (gray-100) as a visual group indicator. Content reveals with height animation (max-height transition, 200ms). The toggle label changes from "Advanced options" → same text (or collapses with − icon). Never use "Show more" as a label — name the content being revealed.
Do Default all advanced panels to collapsed. Name the hidden content ("Advanced options", "Cost breakdown", "API configuration"). Remember expand/collapse state per user for returning visits. Indent expanded content with a left border to visually group it.
Don't Hide critical information behind progressive disclosure — the path to the first pipeline run must be fully visible. Use accordions for content the user needs to compare (use tabs instead). Nest expandable sections more than 1 level deep.

7. Error Recovery for New Users

Errors are terrifying for first-time users. They don't have the mental model to diagnose what went wrong. Every error state must explain what happened, why, and exactly how to fix it.

Hulick: "Anytime there's an opportunity to increase the likelihood that users are successful — that's onboarding." — Error recovery is onboarding.

A. Failed Step — First-Time User View

When a pipeline step fails, the error card must be self-sufficient. A new user should be able to recover without documentation or support.

Competency-model-s1 FAILED (INFERENCE)
Fetch failed The model couldn't retrieve the upstream data from "Job-tasks." This usually means the upstream step timed out or returned incomplete output.
How to fix 1. Check if "Job-tasks" completed successfully.
2. If it did, click "Restart" below to re-run this step.
3. If not, restart "Job-tasks" first, then retry this step.
Structure: (1) Error title — the raw error string for technical users. (2) Plain-language explanation — what this means in context. (3) "How to fix" box — numbered steps on white background, visually separated from the error. (4) Actions — "Restart" as primary, "View logs" as secondary for advanced users. This mirrors Hulick's principle: guide users toward their next successful action, don't just report the failure.

B. Seed Upload Errors

Invalid seed format
Expected JSON with a "role" field at the root. The uploaded file is missing this field.
Do Every error has three parts: what happened, why, and how to fix it. Use numbered steps for multi-step recovery. Link to relevant docs or guides from error states. Show the raw error string for technical users but lead with the plain-language explanation.
Don't "Something went wrong." "An unexpected error occurred." "Error 500." — these tell the user nothing. Error states that only say "Restart" without context. Red text without a white-background recovery box (the contrast makes the fix hard to read).

8. Returning User Nudges

Onboarding doesn't end at first use. Users who don't return within 48 hours are at risk of churning. These patterns re-engage users who started but didn't finish.

Hulick: "People dislike losing things more than they like getting things. The faster you give someone something they're afraid of losing, the more likely they'll continue." — Loss aversion as a retention tool.

A. Resume Banner

Shown at the top of the dashboard when a returning user has an incomplete pipeline or unfinished getting-started items.

Resume your CMO pipeline
4 of 9 steps completed · last active 2 hours ago
Trigger: User returns with an in-progress pipeline. Show the most recent incomplete pipeline. "Continue" navigates to the pipeline detail page. Dismiss removes for this session; re-shows on next login if pipeline is still incomplete. Include time context ("2 hours ago") to leverage recency.

B. Lifecycle Email Triggers

Trigger Timing Subject / CTA Signed up, no seed uploaded +24 hours "Your first pipeline is 2 minutes away" → Upload seed Seed uploaded, pipeline not run +4 hours "Your seed is ready to run" → Start pipeline Pipeline failed, not retried +24 hours "Your CMO pipeline needs attention" → View pipeline First pipeline complete, no return +48 hours "Your competency model is ready" → Review output Inactive 7 days +7 days "New: batch pipeline runs" → What's new
Rules: Each email is behavior-triggered, not time-based. If the user takes the action before the email fires, cancel it. Max 1 email per 48 hours. Subject lines describe the user's outcome, not the product's features. Every email has exactly 1 CTA button. Plain-text alternative included. Unsubscribe link prominent.
Do Behavior-triggered emails that reference the user's specific data ("Your CMO pipeline"). Resume banners that show progress context. Time-since-last-activity ("2 hours ago") to leverage recency.
Don't Generic drip campaigns on a fixed schedule. More than 1 email per 48 hours. "We miss you!" tone. Emails that list features the user hasn't tried — focus on the outcome they started pursuing.
