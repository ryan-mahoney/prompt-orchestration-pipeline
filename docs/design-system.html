Prompt Pipeline
Design System

A functionalist design reference for building clear, data-dense interfaces. Every element justifies its presence by aiding cognition or task completion.

v1.0 March 2026 Philosophy: Form Follows Data

0. Design Principles

Non-negotiable rules that govern every design decision in this system.

Data-Ink Ratio

Every pixel must contribute to understanding. If you can remove an element without losing meaning, remove it. No decorative gradients, blobs, shadows, or textures that don't encode data.

Top Tasks First

Identify the 5 reasons a user visits any page. Make those actions the primary visual focus. Everything else is secondary or removed.

Color Encodes Meaning

Color is never decorative. Green = success/done. Red = error/failed. Violet = interactive/link. Gray = neutral/pending. Every color choice must map to a semantic category.

Plain Language

Omit needless words. No marketing fluff, no intensifiers, no corporate happy talk. Button labels are verb + noun. Error messages explain what happened and how to fix it.

7. Forms

Every field is a cost. Single-column layout, max-width 640px, top-aligned labels. Validate on blur. Mark optional fields, not required ones.

Do Input height: 40px. Focus: violet ring (3px). Errors: red border + message below. Labels: always visible, concise nouns, sentence case. One primary action.
Don't Placeholders as labels. Multi-column forms (except coupled fields like city/state). Disabled submit to hide errors. Color alone for error states.

13. Accessibility

Accessibility is not optional. These are hard requirements, not aspirational guidelines.

Contrast

Body text on white: minimum 4.5:1 (AA). Prefer 7:1 (AAA). Gray-700 (#374151) on white passes AA. Gray-500 (#6b7280) on white passes AA for large text only — use for metadata, not body.

Keyboard

Every interactive element must be operable via keyboard. Visible focus rings (2px solid accent-700, 2px offset). Tab order follows visual order. No keyboard traps.

Semantic HTML

<button> for actions, <a> for navigation. Proper heading hierarchy (never skip h2 → h4). <table> with <thead> for data tables. aria-sort on sortable columns.

Touch Targets

Minimum 44×44px for all interactive elements. Buttons, links, checkboxes — no exceptions. Small buttons (32px visual) must have 44px hit area via padding.

Status Indicators

Never color alone. Every status badge pairs color with text label. Error states pair red border with error message text. Icons include aria-label or sr-only text.

Forms

Every input has a programmatic <label>. Error messages are linked via aria-describedby. Focus first error on submit. Works at 320px viewport width.

14. Motion

Motion is feedback, not decoration. Transitions confirm state changes. No animations for "delight" that slow the user down.

Fast (100ms)Hover color changes, button press, focus ring appearance Normal (150ms)Border color transitions, input state changes, badge appearance Slow (250ms)Panel expand/collapse, dropdown open, content reveal Easingcubic-bezier(0.4, 0, 0.2, 1) — Material standard easing for all transitions
Do Instant state feedback on click. Smooth hover transitions. Respect prefers-reduced-motion.
Don't Page transitions. Loading spinners slower than the data fetch. Bounce, elastic, or spring animations. Parallax scrolling.

17. Copywriting Rules

Text should be invisible so the message shines through. Objective, helpful, precise — like a technical manual written by a human.

Write this

"Pipeline failed at step 4."

"Call volume is high."

"3 steps complete. 6 remaining."

"To start, upload a seed file."

Not this

"Oops! Something went wrong."

"We're experiencing high call volumes."

"Your pipeline is making great progress!"

"Welcome! In order to begin the process of..."

Omit needless words. "In order to" → "To." "Currently experiencing" → cut it.

Active voice. "The system saves the file" not "The file is saved by the system."

No intensifiers. Ban: very, really, absolutely, cutting-edge, revolutionary, game-changing.

Front-load. Key word first in every label, heading, and sentence.

Pre-Launch Checklist

Run this scan on every page before shipping.

☐ Grid visible? — Alignment is obvious without turning on guides.
☐ Data-ink ratio high? — Can I erase this element without losing meaning?
☐ Color functional? — Does every color encode status, category, or interaction?
☐ Text skimmable? — Headers, short paragraphs, front-loaded keywords?
☐ Adjectives deleted? — Is the language plain and direct?
☐ One primary CTA? — Only one green/primary button per view?
☐ Buttons start with verb? — Verb + noun, outcome predictable?
☐ Accessible? — Contrast ≥ 4.5:1, keyboard operable, semantic HTML, 44px targets?
☐ Empty state exists? — Every view has a zero-data state with a CTA?
☐ Error states exist? — Errors explain what + how to fix, not just "something went wrong"?
